Project Understanding Document
Last Updated: 2026-03-29

1) What this project is
- This is a static frontend demo for a restaurant workflow:
  Table QR -> Customer Ordering -> KOT Queue -> Billing.
- Stack is plain HTML/CSS/JavaScript (no backend, no framework, no build step).
- Runtime persistence is browser localStorage.

2) Current file map
- index.html: Operator dashboard with 3 tabs (Customer simulation, KOT, Billing).
- script.js: Operator logic (dispatch, KOT queue, billing, paid history, sync).
- style.css: Operator styling.
- customer.html: Customer phone ordering UI.
- customer-view.js: Customer ordering logic, offers/upsell logic, cart/order sync.
- customer-view.css: Customer styling.
- data.js: Shared cafe data model (brand, tables, full menu, offer bundles).

4) Functional behavior now
A. Customer ordering
- Opens as customer.html?table=Tn from operator QR links.
- Customer sees brand, table id, live offer cards, searchable/sortable menu, categories.
- Customer can:
  - add/remove normal items
  - add deal bundles from offer cards
  - accept quick upsell suggestion near checkout
- Cart automatically applies active offer pricing and calculates discount.
- On place order:
  - order is saved with status "Queued"
  - applied offer metadata and pricing snapshot are persisted when relevant
  - customer sees ETA and savings message (if discount applied)

B. KOT terminal
- Shows non-paid tickets with statuses:
  Queued, Preparing, Ready.
- Status cycle remains:
  Queued -> Preparing -> Ready -> Queued.
- Queue stats:
  active tickets, average wait, next ready candidate.
- Ticket cards can include applied-offer summary.

C. Billing counter
- Fetch by table number.
- Merge all active table tickets.
- Compute and show:
  itemized lines + subtotal + offer savings (if any) + GST 5% + final amount.
- Mark-as-paid moves tickets to Paid and stores paid bill history with final amount.

5) Data model snapshot
- Shared data source:
  - window.APP_DATA.brand
  - window.APP_DATA.tables
  - window.APP_DATA.menu
  - window.APP_DATA.offers
- localStorage keys:
  - kot_demo_orders_v1
  - kot_demo_paid_bills_v1
- Order model (current practical shape):
  - { id, tableId, status, createdAt, items: [{id, qty}], pricing?, appliedOffers? }
- Paid bill model:
  - { tableId, ticketCount, amount, when }

6) UX state
- Customer page is now more conversion-oriented:
  - direct offer actions
  - clear savings message
  - reduced cognitive load (removed non-essential controls)
- Operator page remains functionally aligned with customer-side pricing and offers.
- Responsive behavior retained for mobile/desktop.

7) Known gaps / risks
- No backend/database/authentication yet.
- No real QR scanner integration (QR behavior is simulated by table links).
- No payment gateway integration.
- No admin UI yet to edit menu/offers dynamically in-browser (currently in data.js).
- No analytics pipeline yet for conversion metrics (impressions/clicks/acceptance/AOV).
- KOT state loop (Ready -> Queued) may not match real kitchen operations.

8) How to run now
- Open index.html in a browser.
- Use "Open Customer View" to open customer.html for a table.
- Place orders and observe KOT/Billing updates via localStorage cross-tab sync.

9) Suggested next steps
1. Build an owner "Menu & Offers Manager" screen to edit data.js content via UI.
2. Add offer analytics counters (view/click/add/order-using-offer).
3. Introduce backend + realtime channel (WebSocket/SSE) for true multi-device operation.
4. Tighten workflow state machine for production-like kitchen flow.
