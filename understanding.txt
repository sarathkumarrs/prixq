Project Understanding Document
Generated on: 2026-03-28

1) What this project is
- This is a static frontend demo for a restaurant flow: Table QR -> Customer ordering -> Kitchen Order Ticket (KOT) queue -> Billing by table.
- It is built with plain HTML, CSS, and vanilla JavaScript (no backend, no framework, no build step).
- Data persistence is browser-side only using localStorage.

2) Current file map
- index.html: Main operator demo screen with 3 tabs (Customer simulation, KOT Terminal, Billing Counter).
- script.js: Core logic for the operator screen (cart, order dispatch, KOT status, billing, paid history, localStorage sync).
- style.css: Styling for the operator screen.
- customer.html: Mobile-style customer page opened via table-specific link/query param.
- customer-view.js: Customer-facing logic (menu selection, place order, offers, ticket status, localStorage sync).
- customer-view.css: Styling for customer page.

3) Functional flow already built
A. Customer / QR simulation
- Table selection supports T1-T4.
- Each table has a table-specific offer.
- Menu has 6 items with category, price, and prep-time metadata.
- Customer can add/remove quantity and place order.
- New orders are created with status "Queued" and auto-generated ID pattern KOT-XXXX.

B. KOT terminal behavior
- Displays active tickets with statuses: Queued, Preparing, Ready (Paid tickets hidden from KOT queue).
- "Move Status" cycles status in sequence:
  Queued -> Preparing -> Ready -> Queued.
- KOT stats shown:
  - Active tickets count
  - Average wait
  - Next ready candidate
- ETA is computed from max prep time in ticket + queue position weighting.

C. Billing behavior
- Billing is fetched by table number (select table).
- Merges all non-paid tickets for selected table into one bill view.
- Shows itemized totals + GST (5%) + final amount.
- "Mark As Paid" marks table's active tickets as Paid and stores a paid-bill history entry.

D. Customer phone view behavior
- Opens with query params, e.g. customer.html?table=T1 and optional ?date=YYYY-MM-DD.
- Shows table offer + daily/special campaign offers.
- Shows active ticket count and ticket statuses for selected table.
- Uses the same orders localStorage key, so customer page and operator page stay in sync across tabs.

4) Data model and persistence
- localStorage keys:
  - kot_demo_orders_v1
  - kot_demo_paid_bills_v1
- Seed/demo data:
  - Two default tickets are auto-seeded when storage is empty.
- Important structures:
  - Order: { id, tableId, status, createdAt, items: [{id, qty}] }
  - Paid bill history: { tableId, ticketCount, amount, when }
- Cross-tab synchronization:
  - Both pages listen to window.storage events and re-render when storage changes.

5) UX/status of implementation
- UI is responsive for desktop and mobile breakpoints.
- Visual system is consistent (same font and color family across both pages).
- This is a strong demo/prototype stage, not production-ready architecture.

6) Gaps / risks identified
- No backend/API/database; all data is local to one browser profile.
- No authentication/roles (any user can access billing/kitchen actions).
- No real QR generation/scanning integration; QR is simulated UI.
- No printer/KOT slip output integration.
- No payment gateway integration.
- No order edit/cancel/split/partial-pay flows.
- No inventory or kitchen station routing.
- Billing inconsistency risk:
  - Billing screen shows GST in final amount,
  - but paid bill history amount currently stores subtotal-only (without GST).
- Status loop includes Ready -> Queued, which may be intentional for demo but is operationally unusual.

7) How to run/use now
- Open index.html in a browser.
- Use "Open Customer View" button to open customer.html for selected table.
- Place orders from either screen and watch KOT/Billing update via localStorage.

8) Suggested next build priorities
1. Introduce a backend (orders, tables, menu, billing) with proper persistence.
2. Define state transitions with stricter workflow (Queued -> Preparing -> Ready -> Paid, no loop).
3. Fix billing amount consistency (store and display same final total definition).
4. Add auth + role-based screens (Customer, Kitchen, Cashier, Admin).
5. Add order lifecycle features (cancel, merge/split bills, reprint, audit logs).
6. Add real-time channel (WebSocket/SSE) instead of localStorage events.
